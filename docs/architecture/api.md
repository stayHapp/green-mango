# API 设计

本文档记录第一版概念性 API 设计。当前只实现了健康检查接口，业务接口暂未实现。

## 健康检查

- 方法：`GET`
- 路径：`/api/health`
- 用途：确认后端服务可用。
- 权限：公开接口，不返回敏感信息。

## 会议列表

- 方法：`GET`
- 路径：`/api/meetings`
- 用途：获取会议列表。
- 权限：后台接口，第一版应要求管理员权限。

## 创建会议

- 方法：`POST`
- 路径：`/api/meetings`
- 用途：创建会议。
- 权限：后台接口，第一版应要求管理员权限。

## 获取会议详情

- 方法：`GET`
- 路径：`/api/meetings/{meeting_id}`
- 用途：获取会议基本信息。
- 权限：后台详情需要管理员权限；前台公开详情可后续拆分公开接口。

## 修改会议

- 方法：`PATCH`
- 路径：`/api/meetings/{meeting_id}`
- 用途：修改会议基本信息。
- 权限：后台接口，第一版应要求管理员权限。

## 获取报名字段

- 方法：`GET`
- 路径：`/api/meetings/{meeting_id}/registration-fields`
- 用途：获取某个会议的报名字段配置。
- 权限：前台报名页可读取公开字段；后台可读取完整字段配置。

## 配置报名字段

- 方法：`PUT`
- 路径：`/api/meetings/{meeting_id}/registration-fields`
- 用途：整体保存某个会议的报名字段配置。
- 权限：后台接口，第一版应要求管理员权限。

## 提交报名

- 方法：`POST`
- 路径：`/api/meetings/{meeting_id}/registrations`
- 用途：提交某个会议的报名信息。
- 权限：前台公开接口，但必须做输入校验、字段校验和频率限制规划。

## 查看报名记录

- 方法：`GET`
- 路径：`/api/meetings/{meeting_id}/registrations`
- 用途：查看某个会议的报名记录。
- 权限：后台接口，第一版应要求管理员权限。

## API 设计原则

1. 前端校验只改善体验，后端必须做完整校验。
2. 后台接口必须进行权限判断。
3. 动态字段提交时，后端必须根据当前会议字段配置校验字段类型和必填规则。
4. API 变更后必须同步更新本文档。
