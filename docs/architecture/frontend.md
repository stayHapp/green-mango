# 前端架构

本文档记录三端前端当前结构与逐步联调边界。

## 页面结构

当前页面按三个客户端组织：

- 管理员端：统一登录、会议列表和会议详情。
- 工作人员端：统一登录、负责会议和签到工作台。
- 嘉宾端：会议入口登录、参会会议、个人资料和二维码。

## 组件规划

可逐步抽取以下组件：

- 会议基础信息表单
- 报名字段配置表格
- 报名字段编辑器
- 动态报名表渲染组件
- 报名记录表格
- 字段值展示组件

## API 请求封装

`frontend/src/api/` 已提供：

- `client.ts`：从 `VITE_API_BASE_URL` 读取基础地址，默认使用 `http://127.0.0.1:8000/api`。
- 管理员端会议入口链接优先使用 `VITE_PUBLIC_APP_URL` 作为公开前端地址；本地联调配置本机局域网 IP，部署时配置正式域名，避免二维码包含其他设备无法访问的 `localhost`。
- `authStorage.ts`：按管理员、工作人员、嘉宾分别保存用户展示数据、Bearer token 和过期时间。
- `sessions.ts`：三端登录、公开会议入口、嘉宾个人资料和统一退出请求。
- `adminMeetings.ts`：管理员会议列表、创建、详情和修改。
- `adminGuests.ts`：管理员嘉宾列表、新增、完整资料、动态字段读取和二维码批量生成。
- `adminExcel.ts`：管理员嘉宾导入模板、XLSX 名单上传和全量签到表下载。
- `adminStaff.ts`：管理员查询、创建、修改工作人员，以及解除会议授权。
- `adminCheckIns.ts`：管理员读取会议签到统计和签到明细。
- `staffCheckIns.ts`：工作人员读取负责会议、搜索嘉宾、查询签到记录并执行扫码或人工签到。
- `guestMeetings.ts`：嘉宾读取本人可访问的会议列表和会议详情。

授权业务 API 使用 `authorizationConfig(role)` 注入相应客户端 token；FastAPI 的 `detail` 和网络错误由统一函数转换为页面中文提示。

## 路由规划

当前主要路由：

- `/login`
- `/admin/meetings`
- `/admin/meetings/:id`
- `/meetings/:id`
- `/guest/login?meetingId=:id`
- `/guest/meetings`
- `/guest/meetings/:id`
- `/staff/meetings`
- `/staff/meetings/:id/check-in`

管理员与工作人员共用 `/login`，嘉宾必须从独立会议入口进入。

## 状态管理规划

Pinia 当前只保存三端用户和访问会话。登录成功后同步到 localStorage，页面刷新可恢复未过期会话；已过期或结构无效的存储会被自动删除。

退出登录先调用后端撤销 token，再清除 Pinia 与 localStorage。即使后端暂时不可用，本地状态也会清理；嘉宾返回原会议登录入口并清空姓名、手机号。

管理员会议列表、创建、详情和修改已通过 `frontend/src/api/adminMeetings.ts` 使用真实 API。管理员会议详情中的嘉宾列表、新增嘉宾、完整资料、动态字段和二维码生成已通过 `frontend/src/api/adminGuests.ts` 使用真实 API，并在页面内处理加载与错误状态。

管理员端 Excel、工作人员维护和签到统计已接入真实 API。工作人员端负责会议、嘉宾搜索、扫码签到、人工签到和签到记录也已完成真实联调；页面按资源维护局部加载与错误状态，不把业务数据放入全局状态。

会议助手已经接入真实后端 API。管理员可按会议读取并编辑五项功能的通用正文、未发布提醒和发布状态；嘉宾详情页按功能标识读取公开内容。固定入口标题与说明仍由前端维护，正文、提醒和发布状态由后端持久化；未发布响应不包含草稿正文。

嘉宾二维码组件通过真实二维码接口读取 token、有效期和签到状态。存在签到记录时，二维码上覆盖“已签到”标记并显示本地签到时间；刷新页面或重新登录后可恢复服务端状态。

嘉宾“我的会议”和会议详情已接入真实 API。详情页继续复用登录时由个人资料接口返回的固定资料，并组合真实二维码与会议助手接口，形成登录、会议列表、详情和签到状态的完整访问链路。

三端业务页面已不再依赖前端 Mock API。`frontend/src/mock/data.json` 作为早期原型的只读样例数据保留，仅用于演示参考和未来测试夹具，不进入生产运行链路。

管理员会议编辑页通过后端高德代理搜索并选择导航地点，页面只保存地点名称、地址和坐标，不接触高德服务端 Key。嘉宾路线页使用高德 URI 打开手机导航，并将会议助手“路线指引”正文作为到场说明展示。天气和导航共用管理员确认坐标，避免文字地点匹配偏差。

管理员会议详情的签到总数、嘉宾签到状态和签到记录已接入真实统计接口。签到统计作为独立资源加载，失败时只在签到页签展示错误，不影响会议、嘉宾、工作人员和会议助手等其他资料。

签到时间由服务端统一按 UTC 语义返回。工作人员 API 适配层兼容 SQLite 或旧服务返回的无时区 UTC 文本：仅在响应没有 `Z` 或时区偏移时补充 UTC 标识，再由浏览器统一转换为本地时间，避免跨端出现 8 小时时差。
