# 0003 Harness Engineering

## 状态

已接受。

## 决策

本项目采用 Harness Engineering 思路组织 AI 辅助开发。

仓库是项目真实上下文来源，重要设计、范围、架构和执行计划必须沉淀到文件中，而不是只留在聊天记录里。

## 原因

知会会长期借助 AI 辅助开发。为了避免上下文丢失、需求漂移和一次性生成过多不可控代码，需要建立可读、可追踪、可验证的工程协作方式。

## 具体规则

1. `AGENTS.md` 作为 AI 开发地图，保持简洁。
2. 产品和架构设计写入 `docs/`。
3. 复杂任务先写执行计划。
4. 执行计划放在 `docs/exec-plans/active/`，完成后移到 `docs/exec-plans/completed/`。
5. 每次只完成一个明确的小任务。
6. 数据库结构变化同步更新数据库文档。
7. API 变化同步更新 API 文档。
8. 修改完成后说明验证方式。
9. 不一次性生成大量未经确认的业务代码。
10. 新增重大技术决策时，在 `docs/decisions/` 增加记录。

## 影响

项目早期会先建立文档体系和骨架，再逐步初始化前端、后端、数据库迁移和业务功能。这样牺牲少量初始速度，换取后续更稳定的协作效率。
