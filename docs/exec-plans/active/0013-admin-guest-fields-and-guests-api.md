# 管理员嘉宾字段与嘉宾管理 API

## 背景

管理员会议管理 API 已落地，当前 MVP 的下一条核心链路是为会议配置嘉宾字段，并由管理员提前录入嘉宾。前端已使用 Mock 数据验证该流程，后端需要提供可持久化的最小接口。

## 用户确认状态

用户已明确要求继续实现后续 API。本任务只实现管理员侧嘉宾字段配置、嘉宾列表与单个嘉宾录入。

## 目标

- 实现会议嘉宾字段的查询与替换保存。
- 实现已授权管理员的嘉宾列表查询与单个嘉宾创建。
- 为新嘉宾生成不含个人信息的随机二维码 token。
- 覆盖会议授权、字段 key 唯一性、必填嘉宾信息和二维码 token 生成测试。

## 不在本次范围内的内容

- 不实现 Excel 导入、嘉宾编辑/删除、嘉宾动态字段值写入。
- 不实现嘉宾登录、二维码图片生成、二维码过期校验、签到或导出。
- 不接入 Vue 前端真实请求。

## 技术决策与待确认项

- 嘉宾字段配置使用 `PUT /api/admin/meetings/{meeting_id}/guest-fields` 全量替换；若当前会议已有嘉宾动态字段值，接口拒绝替换，避免无意丢失历史数据。
- 嘉宾创建先覆盖姓名、手机号、单位、职务、身份标签和座位号等固定 MVP 字段；动态 `guest_values` 留给后续字段值 API。
- 使用 Python `secrets` 生成随机二维码 token，且数据库唯一索引作为最终冲突保护。

4. 更新架构文档并运行完整测试和迁移验证。

## 验收标准

- 已授权管理员可以读取和保存嘉宾字段配置。
- 未授权管理员无法读取或修改会议嘉宾字段、嘉宾列表。
- 已授权管理员可创建姓名与手机号齐全的嘉宾，响应含随机二维码 token。
- 同一会议的嘉宾字段 key 不能重复。
- 已存在嘉宾字段值时，字段全量替换被安全拒绝。

## 验证方式

- 在 `backend` 目录使用 Python 3.12 虚拟环境运行 `pytest`。
- 使用临时 SQLite 数据库运行 `alembic upgrade head`。
- 运行 `git diff --check`。

## 风险与注意事项

- 当前 `X-Admin-Id` 仅限本地开发和测试，不能作为生产鉴权方案。
- 预留的动态字段值将在后续单独实现，当前不得误导为嘉宾录入已支持任意字段值。
