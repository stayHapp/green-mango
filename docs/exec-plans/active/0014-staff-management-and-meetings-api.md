# 工作人员管理与负责会议 API

## 背景

管理员已可创建会议和嘉宾，下一步需要创建现场工作人员并将其授权到会议；工作人员端随后才能获取自己负责的会议并进入签到流程。

## 用户确认状态

用户已明确要求继续实现后续 API。本任务只实现工作人员创建、会议授权与工作人员会议列表。

## 目标

- 管理员可创建工作人员账号并自动授权其负责当前会议。
- 管理员可查看会议已授权工作人员。
- 工作人员可查询自己负责的会议。

## 不在本次范围内的内容

- 不实现正式工作人员登录、密码设置、会话令牌或签到 API。
- 不实现工作人员编辑、停用、解除授权或跨会议批量授权。

## 技术决策与待确认项

- 管理员创建工作人员时接受唯一账号、姓名和可选手机号；由于认证任务未开始，密码哈希暂保存不可用于登录的随机占位值。
- 工作人员会议列表临时使用 `X-Staff-Id` 请求头，并验证用户角色为 `staff` 与启用状态；正式认证上线后必须替换。
- 若同一账号已存在且角色为工作人员，则只补充当前会议授权；管理员账号不能转换为工作人员。

## 分步计划

1. 定义工作人员创建、响应结构和开发期工作人员身份依赖。
2. 实现工作人员创建、会议授权、工作人员列表与负责会议列表服务、路由。
3. 覆盖账号复用、授权过滤、角色校验和未授权访问测试。
4. 更新文档并执行完整验证。

## 验收标准

- 已授权管理员可创建工作人员并授予当前会议权限。
- 重复创建同一工作人员账号不会产生重复用户或重复授权。
- 管理员可查看当前会议工作人员。
- 工作人员只能查看自己的负责会议；非工作人员或停用账号被拒绝。

## 验证方式

- 在 `backend` 目录使用 Python 3.12 虚拟环境运行 `pytest`。
- 使用临时 SQLite 数据库运行 `alembic upgrade head`。
- 运行 `git diff --check`。

## 风险与注意事项

- 本任务的随机密码哈希占位值不能用于登录，工作人员登录必须在正式认证任务中实现。
