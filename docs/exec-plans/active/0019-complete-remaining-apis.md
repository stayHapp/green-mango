# 完成剩余全部 API

## 背景

当前后端已实现会议、嘉宾、工作人员、签到和统计的核心接口，但仍使用可伪造的开发期身份请求头，并缺少 Excel 导入导出、资源维护、多管理员管理、动态字段值和嘉宾补充报名等接口。

## 用户确认状态

用户已明确要求先完成所有 API 开发，再进行前后端联调。

## 总体目标

- 完成管理员、工作人员和嘉宾的安全登录、会话校验与退出。
- 补齐嘉宾动态字段、编辑停用、登录规则、批量二维码等管理 API。
- 补齐工作人员维护、多管理员授权与解除 API。
- 完成 Excel 嘉宾导入模板、导入与签到表导出。
- 完成嘉宾补充报名入口及必要的数据结构。
- 增加 CORS 配置和完整自动化测试，移除业务接口对开发期 ID 请求头的依赖。

## 实施阶段

1. 安全认证与会话：密码哈希、会话表、三端登录、Bearer token、退出登录。
2. 资源补全：嘉宾动态字段值、嘉宾维护、工作人员维护、多管理员、登录规则和二维码批处理。
3. Excel：模板下载、名单导入、签到表导出及工作簿往返验证。
4. 嘉宾补充报名、CORS、文档整理、迁移和完整回归测试。

## 不在本次范围内的内容

- 微信、短信验证码、支付、邮件、离线签到和复杂多租户权限。
- 前端真实 API 接入。

## 验收标准

- `docs/architecture/api.md` 中 MVP 接口均有实现或明确取消理由。
- 三端业务接口使用服务端验证的 Bearer token，不再依赖客户端直接声明用户 ID。
- Excel 导入导出可由自动化测试真实读取并核验内容。
- Alembic 可从空库升级到最新版本，pytest 全部通过。

## 验证方式

- Python 3.12 下运行完整 `pytest`。
- 临时 SQLite 数据库执行 `alembic upgrade head`。
- 运行 `git diff --check`。

## 风险与注意事项

- 任务跨度较大，按以上阶段逐项提交可验证结果；任一阶段失败不会跳过验证继续扩展。
- Excel 文件由服务端运行时库生成，需纳入后端依赖并锁定兼容版本范围。
