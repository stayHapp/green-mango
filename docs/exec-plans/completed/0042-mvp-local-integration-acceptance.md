# MVP 本地全链路联调与验收

## 任务名称

MVP 本地全链路联调与验收。

## 背景

管理员端、嘉宾端、工作人员端及核心后端 API 已完成基础接入。当前需要按照真实会议业务顺序完成一次本地全链路联调，识别并修复跨端状态、权限、签到和统计中的问题，为 MVP 阶段验收收口。

## 用户确认状态

用户已明确要求继续完成 MVP 联调，可以执行本计划内的本地测试、测试数据初始化、缺陷修复和文档更新。

## 目标

- 验证管理员创建和配置会议、维护嘉宾及工作人员的业务链路。
- 验证嘉宾身份核验、会议信息、会议服务、个人资料和签到二维码。
- 验证工作人员搜索嘉宾、手工或扫码签到、重复签到与权限约束。
- 验证管理员签到统计和全量导出结果与签到状态一致。
- 修复联调发现的 MVP 范围内缺陷，并完成自动化回归。

## 不在本次范围内的内容

- 微信、短信、支付、通知、文件上传和多租户。
- 多次签到、撤销签到、离线签到和分会场签到。
- 新增会议服务类型或高级页面装修。
- 使用真实嘉宾资料或修改测试服务器数据。

## 技术决策与待确认项

- 使用 `app.scripts.seed_dev` 生成可重复的本地测试账号和会议数据。
- 浏览器交互与 API 检查结合，避免只验证页面或只验证接口。
- 扫码能力无法使用真实摄像头时，以二维码令牌接口和手工签到共同验证核心签到规则。
- 本轮发现的缺陷限定在 MVP 范围内直接修复；超出范围的问题记录为后续任务。

## 分步计划

1. 确认本地前后端服务、数据库迁移和联调测试数据。
2. 运行后端测试、前端构建、代码差异检查和核心健康接口。
3. 按管理员工作流验证会议、嘉宾、工作人员和会议服务配置。
4. 按嘉宾工作流验证登录、首页、服务详情、个人资料和二维码。
5. 按工作人员工作流验证搜索、签到、重复签到、过期和权限规则。
6. 回到管理员端验证统计和导出，并核对跨端状态一致性。
7. 修复问题、补充测试和文档，运行全量回归后归档计划。

## 验收标准

- MVP 三端核心页面均可使用真实后端数据完成业务操作。
- 一个嘉宾在一个会议内只能成功签到一次，重复操作不产生第二条记录。
- 嘉宾、工作人员和管理员看到的签到状态保持一致。
- 未发布会议服务不泄露正文，已发布内容可由嘉宾查看。
- 管理员导出内容与当前嘉宾及签到数据一致。
- 后端测试、前端生产构建和代码差异检查全部通过。

## 验证方式

- `./scripts/check_all.sh`
- `cd frontend && npm run build`
- `git diff --check`
- 浏览器逐端联调管理员、嘉宾和工作人员页面。
- 必要时使用本地 API 请求核对数据库状态与导出文件。

## 风险与注意事项

- 联调会改变本地开发数据库中的演示会议、发布状态和签到记录，不影响测试服务器或正式数据。
- `seed_dev` 会重置固定演示账号密码，但不会删除其他本地数据。
- 天气和地点搜索依赖本地第三方服务配置，外部服务异常应验证降级提示而不是阻断核心闭环。
