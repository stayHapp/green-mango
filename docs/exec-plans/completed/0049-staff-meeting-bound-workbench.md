# 工作人员会议专属入口与移动签到工作台

## 背景

工作人员端入口由会议公开首页提供，因此进入时已经确定会议。当前实现仍要求工作人员登录后进入负责会议列表，再次选择会议，并且扫码、人工签到和签到记录集中在一个较长页面中，不符合现场单任务操作方式。

## 目标

- 在会议公开首页底部提供当前会议专属工作人员入口。
- 工作人员登录成功后校验当前会议授权并直接进入该会议签到工作台。
- 工作台使用与嘉宾端一致的墨绿色移动端主题。
- 使用底部导航区分扫码签到、手动签到和签到记录。
- 手动签到支持搜索嘉宾、核对嘉宾资料和二次确认。

## 范围

- 调整前端路由、会议公开首页、工作人员登录页和签到工作台。
- 继续复用现有工作人员登录、会议授权、嘉宾搜索、扫码签到、人工签到和签到记录 API。
- 不修改数据库和后端 API。
- 保留旧工作人员会议列表路由作为兼容入口，但会议专属入口不再经过该列表。

## 实施步骤

1. 新增会议专属工作人员登录路由和页面。
2. 在会议公开首页增加工作人员入口。
3. 登录后查询工作人员授权会议，校验当前会议并直达签到工作台。
4. 重构签到工作台布局和三种底部导航状态。
5. 为手动签到增加搜索、选择、核对和确认流程。
6. 更新产品与前端架构文档。
7. 执行前端构建、浏览器交互验证和效果图对比。

## 验证方式

- 从 `/meetings/:id` 进入工作人员登录页，确认会议 ID 保持绑定。
- 使用已授权工作人员账号登录，确认直接进入 `/staff/meetings/:id/check-in`。
- 验证扫码区、手动搜索结果、嘉宾确认、签到记录和底部导航。
- 验证无权限会议不会进入工作台。
- 执行 `npm run build` 与 `git diff --check`。

## 完成结果

- 会议首页工作人员入口已绑定当前会议，登录后直接进入签到工作台。
- 工作台已完成扫码签到、手动搜索与核对、签到记录三种操作模式。
- 手动签到已实现按姓名、手机号、单位或座位号搜索嘉宾，再核对信息并确认签到。
- 前端构建、差异检查和移动端浏览器交互验证均已通过。
