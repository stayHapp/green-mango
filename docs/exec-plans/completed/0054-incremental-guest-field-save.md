# 嘉宾字段增量保存与数据保护

## 背景

当前嘉宾字段保存采用删除全部字段后重新创建的方式。只要会议中存在任意动态字段值，管理员修改字段名称、必填或展示状态也会收到“不能全量替换”的错误，阻断了安全的日常配置修改。

## 用户确认状态

用户已明确要求开始执行。

## 目标

- 已有字段按稳定 `key` 原位更新，保留字段 ID 和嘉宾字段值。
- 允许修改字段名称、排序、必填、报名展示、个人信息展示和启用状态。
- 允许新增字段和删除没有嘉宾值的字段。
- 删除含值字段、修改含值字段类型时返回具体的数据保护提示。

## 不在本次范围内的内容

- 不提供字段值批量迁移或类型转换工具。
- 不允许直接修改已有含值字段的稳定 `key`。
- 不新增数据库表或迁移。

## 技术决策与待确认项

- PUT 接口继续接收完整目标字段集合，但后端按 `key` 对现有记录执行增量对账，不再删除重建全部字段。
- `key` 作为稳定业务标识；提交新 `key` 等价于新增字段并删除旧字段，因此旧字段有值时仍会被保护。
- 删除和字段类型变化仅在目标字段已有值时阻止，其他元数据修改不受影响。

## 分步计划

1. 将字段保存服务改为按 `key` 增量新增、更新和删除。
2. 增加已有值字段修改、删除和类型变化的接口测试。
3. 更新前端提示和接口文档，移除“全量替换”误导表述。
4. 运行后端测试、前端构建和浏览器真实保存验证。

## 验收标准

- 修改“饮食偏好”字段名称或配置后保存成功。
- 保存前后的字段 ID 和已有嘉宾字段值不变。
- 删除含数据字段时提示具体字段不能删除。
- 后端测试、前端构建和浏览器验证通过。

## 验证方式

- 运行管理员会议 API 测试。
- 运行前端生产构建和差异检查。
- 在真实嘉宾字段页面修改已有含值字段并保存，再刷新确认结果。

## 风险与注意事项

- 必须在变更数据库对象前完成所有破坏性变更校验，避免失败请求留下半完成的会话状态。
- 嘉宾端呈现字段与报名字段配置继续在字段保存成功后同步更新。
