## 目标

- 软停用嘉宾不再进入当前名单、签到统计、工作人员搜索和当前名单导出。
- 后台新增、编辑和详情均能维护或展示已启用的动态嘉宾字段。
- 会议内姓名和手机号完全相同的启用嘉宾只能存在一条，停用后允许重新录入。
- 后端测试、数据库迁移、前端构建和真实浏览器验收全部通过。
- 验收通过后创建一版 Git 提交。

## 不在本次范围内的内容

- 不新增停用嘉宾恢复界面。
- 不实现分页、通知、复杂账号找回或生产运维能力。
- 不改变嘉宾端姓名加手机号的登录规则。
- 不修改与当前阻断问题无关的界面设计。

## 技术决策与待确认项

- 当前名单相关查询统一只读取 `is_active=true` 嘉宾；数据库仍保留停用嘉宾及历史签到数据。
- 使用数据库部分唯一索引保护启用嘉宾的 `meeting_id + name + phone`，兼容 SQLite 和 PostgreSQL，并在服务层提前返回中文业务错误。
- 后台动态字段表单只呈现已启用字段；停用字段历史值保留但不要求新嘉宾填写。
- 当前无待确认项。

## 验收标准

- 删除嘉宾后，该嘉宾不再出现在后台列表、总览人数、工作人员搜索和当前名单导出。
- 配置必填动态字段后，后台新增嘉宾可以填写并保存；编辑后详情可显示最新值。
- 同一会议重复创建相同姓名和手机号返回明确中文错误；停用旧记录后允许重新创建。
- 所有自动化和浏览器检查通过且控制台无业务错误。

## 验证方式

- `backend/.venv/bin/pytest -q`
- 临时 SQLite 数据库执行 Alembic 全量升级、回退和再升级。
- `npm run build`
- 浏览器验证管理员新增、编辑、详情，嘉宾端和工作人员端基础冒烟。

## 风险与注意事项

- 生产历史库若已存在启用的重复身份，迁移创建唯一索引会失败；迁移前需要先检测并给出明确错误，不能静默删除数据。
- 软停用数据仍保留在数据库中，后续若增加恢复功能需重新校验当前启用身份冲突。
- 动态字段编辑不得覆盖未启用字段的历史值。
