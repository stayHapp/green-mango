# 客户端模型

知会当前产品按使用对象划分为三个主要客户端：管理员端、嘉宾端、工作人员端。

## 登录入口

第一版提供两个登录入口：

- 管理与签到入口：管理员和工作人员共用，均使用账号和密码登录。
- 嘉宾入口：嘉宾通过会议入口二维码进入会议专属登录页，并使用姓名和手机号核验身份。

管理员进入管理员端会议管理，工作人员进入工作人员端会议列表或签到工作台，嘉宾进入对应会议的嘉宾页面。

三个客户端均应提供退出登录入口，退出后清除当前客户端会话并返回对应登录页。

## 管理员端

管理员端面向会议组织者和管理人员。

管理员负责：

- 创建会议。
- 编辑会议基础信息。
- 配置嘉宾信息字段。
- 提前录入嘉宾信息。
- 创建工作人员账户。
- 管理会议管理员。
- 查看嘉宾列表。
- 查看签到情况。
- 查看会议相关管理数据。
- 配置并发布会议服务。

一个会议可以有多个管理员。第一版可以先不细分管理员权限，但文档和后续模型需要保留扩展空间。

## 嘉宾端

嘉宾端面向被邀请或已确认参加会议的嘉宾。

嘉宾负责：

- 通过会议专属嘉宾入口进入嘉宾端。
- 查看入口所属的当前会议。
- 查看当前会议相关信息和会议服务。
- 查看个人参会信息展示。
- 展示个人签到二维码。
- 查看签到状态。

嘉宾由管理员提前录入或导入。第一版不提供嘉宾端自主报名入口。

第一版嘉宾从会议专属入口使用姓名和手机号核验当前会议内的身份。该身份不是跨会议账号；后续如需跨会议访问，必须单独设计嘉宾账号体系。

## 工作人员端

工作人员端面向现场签到执行人员。

工作人员负责：

- 通过管理与签到入口进入工作人员端。
- 选择或进入负责的会议。
- 扫描嘉宾二维码。
- 搜索并核验嘉宾。
- 完成扫码签到或手工签到。
- 查看签到结果。
- 处理重复签到、二维码过期、嘉宾不存在等现场情况。

工作人员账户由管理员创建。

## 三端关系

```text
管理员端 -> 创建会议、录入嘉宾、创建工作人员、查看签到
嘉宾端 -> 核验会议级身份、查看当前会议信息和会议服务、展示个人二维码
工作人员端 -> 登录、搜索核验、扫码或手工完成签到
```

第一版产品核心不再是通用报名表，而是会议嘉宾信息管理与现场签到闭环。
