# 客户端模型

知会当前产品按使用对象划分为三个主要客户端：管理员端、嘉宾端、工作人员端。

## 统一登录

第一版前台只提供一个登录页面。

用户输入姓名和手机号后，系统根据已录入的管理员、工作人员、嘉宾信息判断身份，并进入对应客户端：

- 管理员进入管理员端会议管理。
- 工作人员进入工作人员端会议列表或签到工作台。
- 嘉宾进入对应会议的嘉宾页面。

如果同一姓名和手机号匹配多个身份，真实后端需要返回可选身份或按明确权限规则处理；当前 mock 原型按管理员、工作人员、嘉宾顺序匹配。

## 管理员端

管理员端面向会议组织者和管理人员。

管理员负责：

- 创建会议。
- 编辑会议基础信息。
- 配置嘉宾信息字段。
- 提前录入嘉宾信息。
- 创建工作人员账户。
- 管理会议管理员。
- 查看嘉宾列表。
- 查看签到情况。
- 查看会议相关管理数据。

一个会议可以有多个管理员。第一版可以先不细分管理员权限，但文档和后续模型需要保留扩展空间。

## 嘉宾端

嘉宾端面向被邀请或已确认参加会议的嘉宾。

嘉宾负责：

- 通过统一登录进入嘉宾端。
- 查看自己参加的会议。
- 查看会议相关信息。
- 查看个人参会信息展示。
- 展示个人签到二维码。

嘉宾默认由管理员提前录入。系统可以保留嘉宾报名接口作为补充入口，但第一优先级是管理员录入。

嘉宾登录方式由管理员根据已录入的嘉宾字段指定。当前原型使用姓名和手机号作为统一登录验证字段。

## 工作人员端

工作人员端面向现场签到执行人员。

工作人员负责：

- 通过统一登录进入工作人员端。
- 选择或进入负责的会议。
- 扫描嘉宾二维码。
- 完成签到。
- 查看签到结果。
- 处理重复签到、二维码过期、嘉宾不存在等现场情况。

工作人员账户由管理员创建。

## 三端关系

```text
管理员端 -> 创建会议、录入嘉宾、创建工作人员、查看签到
嘉宾端 -> 登录、查看会议信息、展示个人二维码
工作人员端 -> 登录、扫码、完成签到
```

第一版产品核心不再是通用报名表，而是会议嘉宾信息管理与现场签到闭环。
