# 嘉宾端产品规范

## 1. 文档目的与适用范围

本文档定义「知会」嘉宾端的整体体验、信息架构、页面职责、内容优先级、通用状态和 MVP 边界，是后续产品设计、前端实现、接口验收和体验回归的共同依据。

本文档适用于：

- 嘉宾通过会议专属入口完成身份验证后的全部页面。
- 嘉宾查看会议、个人参会信息、签到状态和会议服务的流程。
- 管理员配置内容在嘉宾端的展示规则。
- 移动端优先的页面布局与交互决策。

本文档不定义管理员端编辑表单的具体布局，不替代数据库和 API 文档，也不要求本次调整现有模型、接口或业务代码。

## 2. 产品定位与设计原则

嘉宾端不是单独的“签到页面”，而是一场会议的移动服务首页。签到是核心闭环之一，但不应遮蔽嘉宾获取日程、资料、路线和会务支持的需求。

嘉宾端遵循以下原则：

1. **先确认会议与身份，再提供操作。** 嘉宾首先需要确认自己进入了正确会议，并能识别自己的参会身份。
2. **一个首屏聚焦一组现场任务。** 会议信息独立识别；个人身份与签到可以组合为参会凭证，减少卡片堆叠，但必须保持内容层级清晰。
3. **状态优先于功能入口。** 已签到、未发布、加载失败等状态必须直观，不让嘉宾猜测系统是否正常。
4. **移动端优先。** 关键操作应适合单手操作，图标必须配合文字标签，不使用含义不明确的图标作为唯一提示。
5. **内容不足也要解释下一步。** 未发布时说明当前状态和后续安排，不展示空白页，也不泄露草稿。
6. **实现与规划分开。** MVP 使用现有数据和接口完成稳定闭环，文件、通知、AI 等能力只作为后续扩展。

## 3. 核心用户与典型场景

### 3.1 核心用户

嘉宾端面向已由管理员录入或导入的受邀参会者，典型包括：

- 高中校长、教学副校长、年级负责人等学校管理者。
- 教育专家、高校教师、研究人员和主讲嘉宾。
- 教研员、竞赛负责人、教练和教育评价负责人。
- 教育论坛、研讨、培训和专家讲座的其他受邀嘉宾。

嘉宾的数字产品熟悉程度可能差异较大，因此页面不能依赖隐藏手势、抽象图标或复杂层级。

### 3.2 典型会议

典型场景是持续半天至多天的教育论坛、校长研讨会、学术会议或培训活动，常包含：

- 多个主办、协办单位。
- 多类嘉宾身份和不同座位安排。
- 会前通知、交通与资料准备。
- 现场身份核验和工作人员扫码签到。
- 多环节日程及会务联系需求。

### 3.3 典型任务

嘉宾进入系统后通常需要快速回答：

1. 这是哪一场会议，何时、在哪里举行？
2. 系统识别的我是谁，我的身份和座位是什么？
3. 我是否已经签到，下一步需要做什么？
4. 今天有哪些安排，需要查看哪些资料？
5. 如何到达会场，遇到问题联系谁？

## 4. 会前、会中、会后的体验目标

### 4.1 会前

目标是减少不确定性，帮助嘉宾完成行程和参会准备。

优先提供：

- 会议名称、日期、地点和当前发布状态。
- 个人身份、单位、职务、嘉宾标签和座位等参会信息。
- 会议日程、会议资料、天气提醒、路线导航和联系会务。
- 会议尚未开始或服务尚未发布时的明确说明。

MVP 不要求倒计时、通知推送或个性化行程提醒。

### 4.2 会中

目标是让嘉宾快速完成身份核验、签到和现场信息获取。

优先提供：

- 清晰的签到二维码或已签到状态。
- 当日会议日程和场地信息。
- 会场入口、停车、楼栋、楼层和签到台说明。
- 可立即使用的会务联系方式。

页面刷新后，签到状态必须从服务端恢复；不依赖当前浏览器中的临时状态。

### 4.3 会后

目标是明确会议已经结束，并保留必要的只读信息。

MVP 规则：

- 会议结束后仍可查看已发布的会议信息和会议服务内容。
- 签到二维码明确标记为已过期，不再引导工作人员扫码。
- 不默认提供会议纪要、视频、照片、证书或评价；这些属于后续扩展。

## 5. 嘉宾端信息架构与页面树

```text
会议专属入口
└── 嘉宾登录
    ├── 身份核验失败
    └── 当前会议首页
        ├── 会议概要
        ├── 我的参会身份
        ├── 签到状态与二维码
        ├── 我的完整资料
        ├── 会议服务
        │   ├── 会议日程
        │   ├── 会议资料
        │   ├── 天气提醒
        │   ├── 路线导航
        │   └── 联系会务
        └── 退出登录
```

层级规则：

- 嘉宾身份和会话只属于登录入口对应的当前会议；第一版不提供跨会议列表。
- 嘉宾登录页不重复展示会议名称、时间、地点等会议信息，只保留返回会议首页、登录说明、姓名和手机号表单、登录按钮及报名入口，降低身份核验前的视觉负担。
- 嘉宾登录页使用报名时填写的姓名和手机号完成身份核验；正式界面不得展示示例身份快捷填充，说明文案统一提示登录后可查看专属入场码。
- 姓名和手机号缺失时使用字段级提示并聚焦第一个错误字段；提示文案使用“报名时填写”的口径，身份不匹配等服务端错误显示在提交操作之前。
- “当前会议首页”是一场会议的服务入口，不再额外增加只有功能图标的中间页。
- 五项会议服务进入独立详情页；从详情页返回时进入当前会议的嘉宾首页，并自动展开会议服务抽屉，方便继续选择其他服务。
- 退出登录放在明确的文字操作或“更多”菜单中，不使用电源图标作为唯一表达。
- MVP 不增加底部固定导航；当页面和一级模块继续增长时再评估。

## 6. 嘉宾会议首页

### 6.1 信息层级与模块顺序

首页默认按以下顺序展示：

1. **页面导航区**：当前会议页面不重复展示系统品牌名称；左上角为会议服务菜单，右上角使用退出图标和“退出”文字提供明确的退出操作。
2. **会议概要**：会议名称、起止时间、地点；会议说明只展示摘要，避免占据首屏。
3. **参会凭证**：组合姓名、嘉宾身份、单位、职务、座位与签到状态；未签到时在同一区域展示二维码，已签到时展示成功状态和签到时间。
4. **会议服务菜单**：左上角菜单按钮打开左侧抽屉，在抽屉中提供日程、资料、天气、路线和联系会务。

该顺序保证嘉宾先确认会议和身份，再处理签到及其他服务。常见移动端首屏应同时出现会议服务菜单按钮、会议名称、嘉宾姓名、身份或座位和签到状态。页面不得把发布状态、完整会议说明、完整手机号或大面积装饰区域放在首屏最高优先级。

嘉宾首页的会议名称与公开活动首页使用相同的字体、字号、字重、行高和居中排版；紧凑模式只隐藏时间、地点等会议元数据，不缩小或改变会议名称的视觉规格。

### 6.2 会议概要

必须展示：

- 会议名称。
- 格式统一的起止日期与时间。
- 会议地点。

可选展示：

- 会议说明摘要。
- “查看完整介绍”入口。

规则：

- 多日会议必须明确显示起止日期，不能只展示第一天。
- 地点文字可点击进入路线导航，但不能让点击地点成为唯一的路线入口。
- 当会议时间或地点缺失时显示“待会务确认”，不展示空标签或 `--`。

### 6.3 我的参会身份

优先展示：

- 姓名。
- 嘉宾身份，例如主讲嘉宾、圆桌嘉宾、参会嘉宾。
- 单位和职务。
- 座位号。

隐私与展示规则：

- 姓名、手机号、单位、职务、身份和座位等固定字段仅在管理员选择后呈现；历史会议未配置时默认全部呈现。
- 当前会议首页可以使用“电话”标签展示脱敏手机号，不能展示完整手机号或将电话作为身份主信息。
- 动态字段仅展示管理员选择为嘉宾端呈现的字段。
- 空字段不占据独立行。
- 座位存在时使用明显但不过度夸张的视觉层级，便于现场查找。
- 姓名、身份标签与资料字段之间保留清晰间距；资料正文在手机端不小于 12px，并使用舒展行距，避免单位、职位和电话形成拥挤文本块。

### 6.4 会议服务入口

入口标题统一为：

- 会议日程。
- 会议资料。
- 天气提醒。
- 路线导航。
- 联系会务。

每个入口使用“图标 + 文字标签”。首页不重复展示功能说明；有稳定且与任务直接相关的数据时，才可以补充一行状态，例如天气摘要。入口排序固定，MVP 不允许管理员新增、删除或拖拽排序。

当前会议首页使用左上角菜单按钮收纳会议服务。点击后从左侧打开带遮罩的抽屉，抽屉顶部展示会议名称和关闭按钮，中间纵向展示五项服务，底部提供返回活动首页入口。每项服务必须同时显示图标、文字标签和方向提示；抽屉支持关闭按钮、遮罩和 Escape 键关闭。

## 7. “会议服务”模块定义

### 7.1 产品名称

面向嘉宾的模块统一称为“会议服务”，表达其用途是帮助嘉宾完成参会任务。

现有代码、接口和数据库中的 `meeting assistant`、`assistant-features` 可继续作为内部技术名称。本次产品规范不要求重命名内部模块。

### 7.2 “会议资料”与 `manual` 的兼容关系

面向嘉宾不再把该入口定义为一本已经存在的“会议手册”，而统一展示为“会议资料”。

原因：

- MVP 当前只支持纯文本内容，不支持附件、PDF 或在线预览。
- “会议手册”会让嘉宾预期看到完整手册文件，未发布时容易被理解为系统异常或组织方遗漏。
- “会议资料”可以自然容纳会议介绍、参会须知、会务提示和后续的正式手册。

兼容规则：

- 当前后端固定功能标识仍为 `manual`。
- 当前数据库、API、路由参数和历史数据不变。
- 前端产品标题和文档用语使用“会议资料”。
- 后续支持文件能力时，“会议手册”可作为“会议资料”中的一个子项，而不是恢复为一级入口。

## 8. 会议服务页面规范

### 8.1 通用页面结构

五项服务详情页统一包含：

1. 返回会议首页的导航。
2. 明确的页面标题。
3. 当前内容或状态主体。
4. 必要的更新时间、数据来源或操作说明。
5. 与当前任务直接相关的主要操作。

详情页不重复展示整张嘉宾身份卡和签到二维码。需要切换其他服务时，优先返回会议首页；MVP 不在详情页叠加第二套复杂菜单。

### 8.2 会议日程

内容优先级：

- 日期。
- 开始与结束时间。
- 环节名称。
- 场地。
- 主讲人或说明。

交互规则：

- 多日会议按日期分组，默认定位到当前日期；当前日期无安排时显示最近一个有安排的日期。
- 同一天按开始时间排序。
- 当前 MVP 继续把管理员维护的多行纯文本解析为日期分组和简洁日程卡片；卡片左侧突出时间，右侧展示环节名称以及可选讲者和地点。
- 单行可使用 `标题｜讲者｜地点` 表达可选信息；未填写的讲者或地点不占位。
- 解析失败时按原始段落安全展示，不得丢失正文。
- 未建立结构化日程前，不提供筛选、收藏、冲突检测或个人日程。

### 8.3 会议资料

MVP 可展示：

- 会议介绍。
- 组织机构。
- 报到与参会须知。
- 会务提示。
- 会议纪律或其他管理员发布的纯文本资料。

交互规则：

- 正文按空行或明确标题拆分为可阅读的内容块。
- 已发布内容必须直接可读，不要求再次点击“查看详情”。
- 不展示虚构的文件名、下载按钮或目录。
- 未发布时使用“资料正在准备中”的状态卡，并展示管理员设置的提醒。

后续支持文件后，页面可增加资料列表、文件类型、更新时间、在线预览和下载；这些能力不属于当前 MVP。

### 8.4 天气提醒

内容优先级：

- 管理员确认的会议地点。
- 当前天气。
- 可用预报范围内的近期天气。
- 管理员补充的穿衣、雨具和出行提醒。
- 数据更新时间和数据来源说明。

交互规则：

- 优先使用管理员确认的导航坐标查询天气，历史会议没有坐标时按地点文字降级匹配。
- 会议日期超出供应商预报范围时，明确提示“暂无法提供会议当天预报”，不得用近期天气冒充会议当天数据。
- 天气服务失败不影响其他会议服务，页面保留管理员发布的文字提醒。
- MVP 不提供小时级预警订阅或主动推送。

### 8.5 路线导航

内容优先级：

- 会场名称和地址。
- “打开高德地图导航”主要操作。
- 停车、入口、楼栋、楼层、签到台等到场说明。

交互规则：

- 使用管理员确认的地点名称、地址和坐标生成导航链接。
- 优先调起地图应用，无法调起时回退到高德 H5 页面。
- 知会不在 MVP 内部计算驾车、公交或步行路线。
- 坐标未配置时仍展示会议地址和到场说明，并明确提示导航地点待会务确认。
- 调起第三方地图前不要求额外确认，但按钮应明确将离开当前页面。

### 8.6 联系会务

内容优先级：

- 联系人或会务组名称。
- 联系电话。
- 服务时间。
- 适用事项或紧急联系说明。

交互规则：

- 页面标题和入口统一使用“联系会务”，不使用范围过大的“联系我们”。
- 电话格式合法时可提供拨号操作；当前接口只有纯文本时，至少保证号码可以被选择和复制。
- 有多个联系人时按职责分组，不把所有号码拼成单段文字。
- 不公开与参会服务无关的个人隐私信息。
- MVP 不提供即时通讯、在线客服或分级工单。

## 9. 通用状态规范

状态文案必须说明“发生了什么”和“嘉宾可以做什么”。通用状态如下：

| 状态 | 展示要求 | 可用操作 |
| --- | --- | --- |
| 首次加载 | 在内容区域显示骨架或加载提示，保持标题与返回入口稳定 | 无需操作 |
| 空内容 | 说明当前没有可展示内容，不出现大面积纯空白 | 返回会议首页 |
| 未发布 | 展示功能名称、准备中状态和管理员设置的提醒；不返回草稿正文 | 返回会议首页 |
| 加载失败 | 说明网络或服务暂时不可用，保留已知的页面标题 | 重新加载、返回会议首页 |
| 无权限 | 说明当前身份无法查看该会议或内容，不暗示资源细节 | 返回当前会议登录入口 |
| 会话过期 | 明确登录状态已失效 | 返回原会议登录入口 |
| 会议已结束 | 保留已发布内容只读访问，签到二维码标记过期 | 返回会议首页 |
| 外部服务不可用 | 保留管理员正文和基础信息，说明天气或地图暂不可用 | 重试或使用文本信息 |

补充规则：

- 未发布不是错误状态，不使用红色错误样式。
- 加载失败不能错误显示为“尚未发布”。
- 无权限、不存在和会议未开放可沿用后端统一的资源保护策略，但前端文案要给出安全且可执行的下一步。
- 局部服务失败不得导致整个嘉宾会议首页不可用。

## 10. 签到二维码状态规范

### 10.1 未签到

未签到时，二维码是签到区域的主要内容，必须展示：

- 可识别的完整二维码。
- “请向工作人员出示二维码”说明。
- 会议结束后明确展示二维码失效状态。
- 加载失败时的重试操作。

未签到时不重复展示“当前状态”或“未签到”文字，二维码与操作说明在签到区域居中呈现，直接表达下一步动作。

二维码不得包含姓名、手机号等敏感信息，只编码服务端签发的随机凭证。

### 10.2 已签到

已签到时，成功状态成为主要内容，必须展示：

- 明显的“已签到”状态。
- 签到时间。

当前会议首页在已签到后收起二维码，仅展示成功状态和签到时间，避免继续引导扫码；不需要重复占用首屏展示“无需重复扫码”。其他需要保留核验凭证的独立页面可以继续展示弱化二维码，但必须覆盖明确的已签到标记。

已签到状态必须来自后端唯一签到记录。刷新页面或重新登录后仍应恢复；本阶段不要求 WebSocket（网页套接字）实时推送，工作人员完成签到后可提示嘉宾刷新页面。

### 10.3 过期与异常

- 会议结束后显示“签到已结束”，不再展示可操作的扫码指引。
- 二维码凭证缺失时显示“签到凭证暂不可用”，提供刷新或联系会务建议。
- 重复签到由工作人员端和后端处理，嘉宾端不制造第二条签到记录。

## 11. MVP 范围与后续扩展边界

### 11.1 当前 MVP 与已实现扩展

本规范以仓库当前能力为实现基础：

- 嘉宾通过会议入口使用姓名和手机号登录。
- 嘉宾查看登录入口所属的当前会议、会议概要和个人参会信息。
- 嘉宾展示个人签到二维码并查看服务端签到状态。
- 会议服务固定包含日程、资料、天气、路线和联系五项入口。
- 五项入口由管理员按会议维护正文、未发布提醒和发布状态。
- 日程和资料以纯文本为基础展示。
- 天气通过后端代理读取真实天气，并具备缓存和降级。
- 路线使用管理员确认的高德地点和坐标调起导航。

当前前端已统一使用“会议服务”“会议资料”“联系会务”等产品文案。首页采用紧凑会议概要和参会凭证，会议服务通过左上角菜单按钮打开左侧抽屉；内部技术标识继续保持兼容，不因界面调整修改数据库或 API。

### 11.2 MVP 明确不做

- 会议资料文件上传、PDF 预览、下载和版本管理。
- 结构化日程子表、个人日程、收藏和提醒。
- 嘉宾自主修改个人资料。
- 短信、邮件、微信或系统通知。
- 实时签到状态推送。
- 知会内部地图和路线计算。
- 在线客服、即时通讯和工单。
- 页面自由装修、入口新增删除或管理员排序。
- 跨会议嘉宾账号和“我的会议”聚合列表。

### 11.3 后续扩展

经过独立产品决策后可以扩展：

- 专家介绍和嘉宾阵容。
- 结构化多日程、分会场和个人行程提醒。
- 会议手册、演示文稿等文件资料。
- 会议通知、变更提醒和天气预警。
- 会后纪要、照片、视频、评价和证书。
- AI 会议问答与知识沉淀。

后续扩展不得直接塞入现有通用正文字段；涉及新的排序、权限、文件、状态或生命周期时，应先更新产品、数据库和 API 文档。

## 13. 实现验收清单

后续实现或优化嘉宾端时至少检查：

- 嘉宾能在首屏确认会议、身份和签到状态。
- 首页模块顺序符合本规范，手机端点击区域和文字清晰。
- 产品界面使用“会议服务”“会议资料”“联系会务”等统一名称。
- `manual` 仅作为内部兼容标识，不直接展示给嘉宾。
- 未发布内容不泄露草稿，加载失败不伪装成未发布。
- 已签到后成功状态比二维码更突出，并显示服务端签到时间。
- 会议结束后二维码不再引导签到。
- 天气或地图服务失败不影响其他会议内容。
- 所有返回操作都能回到明确的上一级页面。
- 新增功能没有突破 `mvp-scope.md`，或已先完成对应文档和架构决策。
