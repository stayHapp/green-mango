# 嘉宾端工作流

## 目标

嘉宾端用于让嘉宾通过会议入口二维码进入会议专属页面，完成身份验证，向现场安保或会务人员展示身份信息，并查看会议服务功能。

## 核心流程

1. 嘉宾扫描会议现场或通知中的会议入口二维码。
2. 系统进入嘉宾专属登录页面，并展示会议专属信息。
3. 页面首先展示会议名称、会议时间段、会议地点等基础信息。
4. 嘉宾输入姓名和手机号，核验当前会议内已录入的嘉宾身份。
5. 系统根据姓名和手机号识别身份；如果识别为嘉宾，则进入嘉宾端会议页面并展示身份识别信息。
6. 嘉宾端展示姓名、身份、单位、职务和座位等参会信息；手机号需要展示时默认脱敏。
7. 嘉宾通过“会议服务”查看会议日程、会议资料、天气提醒、路线导航和联系会务等内容。
8. 现场签到功能展示个人签到二维码，由工作人员扫码完成签到。
9. 嘉宾可在页面顶部退出登录，退出后返回嘉宾登录页。

登录后的嘉宾首页信息层级、页面树、状态与交互规范见 `guest-experience.md`。

## 登录方式

嘉宾需要登录。

第一版固定使用姓名 + 手机号登录。后续可允许管理员根据已配置字段自由组合登录验证方式。例如：

- 姓名 + 手机号。
- 手机号 + 嘉宾标签。
- 姓名 + 证件编号。

第一版需要明确一点：嘉宾登录本质上是基于会议内已录入嘉宾信息进行匹配，不等同于完整账号体系。

嘉宾身份只对登录入口所属会议有效。第一版不聚合同一手机号在其他会议中的记录，也不提供跨会议“我的会议”列表。

是否增加短信验证码，后续单独决策。当前文档不默认要求短信验证码。

## 嘉宾可查看内容

嘉宾可以查看：

- 会议名称。
- 会议时间。
- 会议地点。
- 会议说明。
- 个人参会信息。
- 个人签到二维码。
- 嘉宾身份识别信息。
- 会议服务功能菜单。

会议服务内容可逐步增强：

- 结构化会议日程。
- 会议资料中的正式手册下载或在线阅读。
- 更详细的天气提醒。
- 更完整的路线与到场指引。
- 结构化会务联系人。
- 嘉宾证样式。

## 个人信息展示

嘉宾端展示的信息来自管理员录入的嘉宾信息字段。

第一版身份识别信息至少包含：

- 姓名。
- 脱敏手机号，仅在确有核验需要时展示。
- 身份，例如主讲嘉宾、圆桌嘉宾、参会嘉宾。
- 单位、职务、座位等辅助信息。

哪些字段对嘉宾展示，应由管理员配置或由系统默认规则控制。

工作人员在已授权会议的签到工作台中可以查看现场核验所需的完整手机号；该权限不扩展到普通嘉宾页面或无关人员。

## 二维码

每位嘉宾在每个会议下有自己的签到二维码。

二维码有效期按会议结束时间限制。会议结束后，二维码不应再允许签到。

二维码内容不应直接暴露敏感个人信息。后续实现时应使用 token 或签名机制。

### 签到状态标记

- 嘉宾签到成功后，嘉宾端应在个人签到二维码区域明显展示“已签到”标记，避免嘉宾和现场人员重复确认。
- 已签到状态必须来自后端签到记录，页面刷新或重新登录后仍应保持，不使用仅存在于当前页面的临时状态。
- 标记只改变二维码区域的视觉状态，不改变二维码 token、会议结束时间和服务端重复签到校验规则。
- 工作人员完成扫码签到后，嘉宾端刷新页面即可获得最新状态；实时推送暂未实现。

## 暂不处理

第一版暂不处理嘉宾端自主报名、嘉宾自主修改个人信息、短信验证码、通知推送和资料下载。
