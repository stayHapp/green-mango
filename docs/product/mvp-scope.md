# MVP 范围

## 第一版要做什么

第一版 MVP 的目标是验证最核心的会议嘉宾管理与签到闭环：

1. 管理员可以创建会议。
2. 管理员可以查看会议列表和会议详情。
3. 管理员可以修改会议基本信息。
4. 一个会议可以有多个管理员。
5. 管理员可以配置嘉宾信息字段。
6. 管理员可以提前录入嘉宾信息。
7. 管理员可以通过 Excel 批量导入会议嘉宾名单，并获得导入结果与错误提示。
8. 系统可以保留嘉宾报名接口作为补充入口，但第一优先级是管理员录入或导入嘉宾。
9. 第一版嘉宾固定使用姓名 + 手机号登录；管理员可确认该会议启用此登录方式。
10. 嘉宾可以登录嘉宾端。
11. 嘉宾可以查看自己参加的会议、会议相关信息和个人信息展示。
12. 嘉宾端展示个人签到二维码。
13. 嘉宾二维码按会议结束时间过期。
14. 管理员可以创建工作人员账户。
15. 工作人员可以登录工作人员端。
16. 工作人员可以扫码完成嘉宾签到。
17. 工作人员可以按姓名、手机号、单位或座位号搜索嘉宾并核验签到状态。
18. 工作人员网络断开时，系统提示网络异常并禁止新的签到操作。
19. 一个嘉宾在一个会议中只签到一次。
20. 管理员可以查看签到情况并导出全量嘉宾签到表。

## 第一版暂时不做什么

- 微信登录。
- 支付。
- 短信验证码。
- 邮件通知。
- 复杂权限系统。
- 复杂统计分析。
- 文件上传。
- 多租户系统。
- 高级页面装修。
- 多次签到、分会场签到、补签、撤销签到。
- 弱网或离线签到。
- 嘉宾登录字段自由组合。

## MVP 验收标准

1. 可以通过管理员端创建一个会议。
2. 可以为会议配置嘉宾信息字段。
3. 可以录入至少一名嘉宾。
4. 可以通过 Excel 导入嘉宾名单，并识别不合法数据。
5. 可以创建至少一名工作人员。
6. 嘉宾可以使用姓名和手机号登录。
7. 嘉宾可以查看会议信息和个人参会信息。
8. 嘉宾可以展示个人签到二维码。
9. 工作人员可以扫码完成签到。
10. 重复扫码不会重复签到。
11. 会议结束后二维码不能继续签到。
12. 工作人员可以搜索并核验嘉宾的签到状态。
13. 浏览器离线时，工作人员端不能提交新的签到操作。
14. 管理员可以看到嘉宾签到状态并导出签到表。
15. 数据库结构和 API 设计文档与实现保持同步。
