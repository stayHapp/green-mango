# 用户角色

第一版按客户端定义三类核心用户：管理员、嘉宾、工作人员。

## 管理员

管理员是会议组织方或管理人员。

管理员可以：

- 创建会议。
- 查看会议列表和会议详情。
- 修改会议基本信息。
- 配置嘉宾信息字段。
- 提前录入嘉宾信息。
- 确认嘉宾使用姓名 + 手机号核验当前会议身份。
- 创建工作人员账户。
- 查看嘉宾列表。
- 查看签到情况。
- 配置并发布会议服务。

一个会议可以有多个管理员。第一版可以先不细分管理员权限，后续再扩展会议角色和权限模型。

## 嘉宾

嘉宾是被邀请或已确认参加会议的人。

嘉宾可以：

- 登录嘉宾端。
- 查看登录入口所属的当前会议。
- 查看当前会议相关信息和会议服务。
- 查看个人参会信息展示。
- 展示个人签到二维码。
- 查看签到状态。

嘉宾由管理员提前录入或导入。第一版不提供嘉宾端自主报名入口。

第一版嘉宾从会议专属入口使用姓名 + 手机号核验当前会议内的身份，不建立跨会议嘉宾账号。后续可评估登录字段配置或独立账号体系。

## 工作人员

工作人员是现场执行签到的人。

工作人员可以：

- 登录工作人员端。
- 查看自己负责的会议。
- 扫描嘉宾二维码。
- 搜索并核验嘉宾。
- 完成扫码签到或手工签到。
- 查看签到结果。

工作人员账户由管理员创建。第一版工作人员只处理签到，不具备会议编辑和嘉宾信息编辑权限。

## 非核心角色

公开访问者不属于当前 MVP 的业务角色。嘉宾端自主报名是否启用需要后续单独决策，不能复用当前会议级嘉宾身份核验流程直接开放。
